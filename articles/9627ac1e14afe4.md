## はじめに
マイクロフロントエンドを実現する際にはWeb Componentsやiframeといった技術が挙げられますが、今回Web Componentsを使用した際に直面したCORSについて、リバースプロキシを使用して対処した内容をまとめます。

iframeは別オリジン構成を取りやすいという違いがありますが、Web Componentsは親ページと同一のオリジン文脈で動作します。これはつまり、Web Components内でバックエンドに通信する処理がある場合、オリジンの扱いを考慮してAPIを設計する必要が生じることを意味します。

今回検証用に作成したリポジトリは下記です。
@[card](https://github.com/masashi99/cors-reverse-proxy)

## 全体の構成
最初に全体の構成を載せておきます。

![全体の構成](/images/9627ac1e14afe4/all.png)

例として、`/hoge`画面上で`<my-element>`というカスタム要素を使用する場合で考えています。

ブラウザから`/hoge`に対するリクエストを受け付け、HTMLを返却するサーバ(nginx)が存在し、その後ろにカスタム要素の本体を配信するサーバ(nginx)とカスタム要素からリクエストを受けるAPIが存在するような構成となっています。

ポイントの1つ目としては、ブラウザからのリクエストを受け付ける前段のnginxにおいて、下記のように`default.conf`を定義することでリバースプロキシとして動作させ、後方のnginxとAPIに通信を振り分けることでブラウザから見た通信を同一オリジン化している点です。

```nginx:default.conf
server {
    listen 80;
    server_name my-element-proxy;

    location = /hoge {
        root /usr/share/nginx/html;
        try_files /index.html = 500;
    }

    location / {
        proxy_pass http://my-element;
    }

    location /api/ {
        proxy_pass http://api:3000;
        rewrite ^/api/(.*) /$1 break; 
    }
}
```

ポイントの2つ目としては、カスタム要素のURLとAPIのURLをどちらも相対パスで指定することです。相対パスで指定することによって、ブラウザが自動でリバースプロキシへリクエストしてくれます。
これによって、ブラウザから見た際には、常に前段のnginxとの通信で収まるため、CORSエラーを回避できます。

### 処理の流れ
それぞれがどのように連携するかについて解説していきます。

#### 1. ブラウザから`/hoge`へのアクセス
まずはじめにブラウザが`/hoge`というページにアクセスします。

![/hogeにアクセス](/images/9627ac1e14afe4/1.png)

このとき、下記のような`index.html`がブラウザに返却されます。

```html:index.html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>/hogeのページ</title>
    <script type="module" src="/my-element.js"></script>
  </head>
  <body>
    <my-element></my-element>
  </body>
</html>
```

このHTMLのbodyタグには独自コンポーネントとして定義した`<my-element>`の記載と、scriptタグにはその本体である`my-element.js`の取得が含まれています。

Web Componentsで独自のカスタム要素を使用する場合には、`<div>`や`<input>`といった通常のHTMLElementとは異なり、独自定義した`<my-element>`はそのままでは何も意味を成さないため、カスタム要素を定義・登録したJavaScriptファイル（`my-element.js`）が必要となります。

#### 2. `my-element.js`の取得
ブラウザがscriptタグに記載されている`my-element.js`の取得を行います。

![my-element.js取得](/images/9627ac1e14afe4/2.png)

scriptタグのsrcには`/my-element.js`とルート相対パスでリソースを記載しているため、ブラウザは`index.html`の取得元であるnginxにリクエストを行います。

`/my-element.js`のリクエストを受けたnginxは下記の設定により、背後のmy-elementサービス(実態はnginx)へプロキシします。
```diff nginx:default.conf
server {
    listen 80;
    server_name my-element-proxy;

    location = /hoge {
        root /usr/share/nginx/html;
        try_files /index.html = 500;
    }

+   location / {
+       proxy_pass http://my-element;
+   }
}
```

プロキシされたmy-elementサービス(nginx)はmy-element.jsを返却し、最終的にブラウザがそのレスポンスを受け取ります。

#### 3. `<my-element>`のレンダリング
ブラウザが`my-element.js`を実行し、`<my-element>`をレンダリングします。

![api呼び出し](/images/9627ac1e14afe4/3.png)

`<my-element>`内では定義したAPIに対するリクエスト先を`/api/ping`と指定しているため、先ほどと同様にnginxにリクエストを行います。

`/api/ping`のリクエストを受けたnginxは下記の設定により、APIへプロキシします。

```diff nginx:default.conf
server {
    listen 80;
    server_name my-element-proxy;

    location = /hoge {
        root /usr/share/nginx/html;
        try_files /index.html = 500;
    }

    location / {
        proxy_pass http://my-element;
    }

+   location /api/ {
+       proxy_pass http://api:3000;
+       rewrite ^/api/(.*) /$1 break; 
+   }
}
```

:::message
ここではrewriteディレクティブを指定することによって、`/api/ping`へのリクエストを`/ping`へのリクエストとしてAPIへプロキシしています。
このおかげでAPI側はプロキシされることを意識しなくて済みます。
:::

プロキシされたAPIは`/ping`に対するレスポンスを返却し、最終的にブラウザがそのレスポンスを受け取ります。

## カスタム要素自体を開発する際の構成
上記ではカスタム要素からのAPIリクエストをnginx経由でプロキシしていましたが、カスタム要素自体を開発する段階から毎回nginxを起動するのはいささか手間です。

その解決策として、Viteを使用している場合は下記のように`vite.config.ts`を設定することで、APIへのプロキシを行うことができます。

```diff ts:vite.config.ts
import { defineConfig } from 'vite'

// https://vite.dev/config/
export default defineConfig({
  build: {
    lib: {
        entry: 'src/my-element.ts',
        name: 'my-element',
        fileName: 'my-element'
    },
  },
+ server: {
+   proxy: {
+     "/api": {
+       target: "http://localhost:3000", // 起動しているAPIサーバ
+       changeOrigin: true,
+       rewrite: (path) => path.replace(/^\/api/, ""),
+     }
+   }
+ }
})
```

ここでは`/api`で始まるリクエストに対して、`target`でAPIのホストを指定することでプロキシするようにしています。
ここでもURLのrewriteを行うことで、デプロイ時との整合性を保つようにしています。

## まとめ
今回は、Web Componentsを組み込んだ画面で発生するCORSの課題に対して、前段のnginxをリバースプロキシとして配置し、配信先とAPI先を1つのオリジンに集約する方法を紹介しました。

ポイントは、`my-element.js`とAPIの呼び出し先をどちらも相対パスで記述し、nginx側で`location`ごとに適切なバックエンドへ振り分けることです。これにより、ブラウザからは同一オリジン通信として扱われるため、CORSの影響を受けずにWeb Componentsを利用できます。

また、開発時はViteの`server.proxy`を使って同じルーティング方針を再現しておくと、本番構成との差分を減らしながら検証できます。Web Componentsをマイクロフロントエンドに組み込む際の、シンプルで運用しやすい選択肢として参考になれば幸いです。
